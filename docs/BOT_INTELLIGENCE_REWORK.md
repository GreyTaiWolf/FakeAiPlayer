## 2. 北极星与核心原则

目标不是让 Bot 拥有更多零散工具，而是让它在没有 API Key 时也能稳定完成明确任务，在接入
AI 后能理解更自然、更复杂的目标，同时始终由服务器端确定性核心控制实际行为。

```text
玩家命令 / 自主需要
        ↓
本地规则或 AI 意图解释
        ↓
IntentProposal 验证门
        ↓
Mission 仲裁与确定性规划
        ↓
Skill / Task 执行
        ↓
交互姿态与导航
        ↓
Minecraft 世界
        ↓
权威验证、事件、世界模型与记忆
```

必须长期保持以下原则：

1. **无 AI 也能工作**：明确命令、任务规划、寻路、生存中断、恢复与完成验证不能依赖 LLM。
2. **AI 只做高层顾问**：AI 可解释模糊语言、提出 Mission DAG、选择策略、生成声明式建筑意图和自然对话；不得逐 Tick 驾驶。
3. **服务器拥有最终权威**：库存、方块、实体、配方、权限、可达性和完成状态均由真实世界验证器判断。
4. **动作与对话分离**：汇报、打招呼和解释不会终止、替换或假装完成正在执行的任务。
5. **失败必须可分类**：暂停、抢占、取消、世界过期、可重试阻塞和终结失败不能继续混成一个 `FAILED`。
6. **先复现再修复**：每个重要行为修复必须先有可重复的单元测试、GameTest 或受控 harness 场景。
7. **双加载器同语义**：业务逻辑保留在 `common`，Fabric 与 NeoForge 只做注册、生命周期、网络和测试启动适配。
8. **严格生存边界不退化**：AI、寻路或恢复逻辑不能绕过 `CapabilityGate`、隐藏感知限制或建筑确认。

## 3. 无 AI 与接入 AI 的明确边界

### 3.1 无 AI 时必须具备

- 明确命令的本地解析、别名与目标对象绑定；
- Mission 队列、优先级、暂停、继续、取消、替换和追加；
- 配方依赖、材料获取、工具准备和权威后置条件；
- 交互姿态选择、多目标导航、卡住检测、局部重规划；
- 危险感知、战斗/逃跑/进食等本地即时反应；
- 工作记忆、失败分类、检查点与服务器重启恢复；
- 建筑模板、投影确认、材料差分和最终结构验证；
- API 缺失、超时或断开后继续执行已经验证过的 Mission。

### 3.2 AI 接入后只增加

- 对模糊、复合、省略、指代和自然位置描述的理解；
- 多个合法策略之间的高层建议；
- 声明式建筑风格、房间需求和材料偏好；
- 从既往成功经验中检索合适技能或恢复方案；
- 本地恢复策略耗尽后的有限诊断假设；
- 更自然的对话、解释、进度汇报与个性表达。

### 3.3 AI 永远不得执行

- 直接写入逐 Tick 移动、攻击、挖掘或放置输入；
- 自行授予权限、扩大可见范围或启用 operator 能力；
- 运行模型生成的 Java、JavaScript、命令脚本或任意代码；
- 绕过建筑投影确认、区域租约、物资预算或禁止破坏区；
- 根据提示词自行宣布成功；
- 在世界验证失败后用聊天文本覆盖真实状态；
- 把网页、告示牌、书本、容器名称等世界数据当成新的系统指令。

## 6. P0：可复现基线与导航观测

**状态：实现完成 / 提交 `9155c9c` 的 CI #106 通过；发布 baseline 未 pin**

### 范围

- 为导航请求记录 request id、目标、策略、开始位置、解析目标、路径长度、路径成本、耗时和终止原因；
- 为路径执行记录当前节点、目标节点、剩余成本、重规划次数、振荡次数和世界失效原因；
- 建立“树在眼前却绕远路”“固定选树背面”“横移仍被视为进度”“路径中途动态封堵”等失败夹具；
- 将世界场景函数放入无加载器依赖的共享测试 fixture；
- Fabric 与 NeoForge 使用薄注册适配器运行相同场景与阈值；
- 为 NeoForge 配置正式 GameTest server 任务，并以任务退出码、完整服务端日志和 crash report
  作为运行结果；不假定 NeoForge 1.21.3 会生成 Fabric 风格的 JUnit XML；
- 测试资源不得进入生产 JAR；
- 将测试证据绑定提交 SHA、Java/MC/加载器版本、配置与世界种子。

NeoForge 1.21.3 的 GameTest 注册与 `gameTestServer` 配置以
[NeoForged 1.21.3 GameTest 文档](https://docs.neoforged.net/docs/1.21.3/misc/gametest/) 为基线。

当前共享 P0/P1 fixture 精确包含以下 **11 个**场景；两端 wrapper 调用相同的共享方法与 batch id：

1. 加载器启动与测试世界写入；
2. 干燥导航绕开三格宽水区；
3. 活跃的非破坏导航在路径中动态出现实体墙后保持同一 request id、重规划绕行且不破墙；
   初始规划携带的技能级路径约束在动态重规划后仍然有效；
4. Bot 已在树木东、西、南、北任一合法侧时保持当前交互姿态；当前列不满足干燥可站立条件时，
   姿态规划和整树会话均等待安全恢复，不从计算性 snap 的虚假起点启动路径或采伐；
5. 正前方 5 格的可见树保持近侧工作位，实际路径成本不超过最短可交互距离加 2；
6. 近侧被实体方块封堵后改选可站立、可视且可达的工作位；被技能拒绝的树叶支撑短路
   不能遮蔽稍长但合法的地面路线；
7. 石质地基、持久树叶的玩家原木结构和过短木柱不会被识别为自然树；完整四格木柱中经真实
   `BuildAction → useItemOn → BlockItem` 链路放置并由加载器 Mixin 记录的根部正证据，会让整个
   相连组件即使位于天然树冠下也不能进入自动整树；
   无法从地面姿态完整采伐的高树会在砍第一块前整体拒绝；
8. 树冠下的玩家原木横梁、不完整的相邻地基柱，以及有放置证据的同种树顶延长块即使连接到
   带天然土基与非持久叶的橡树，也会让整个混合组件被保守拒绝；
9. 地面可达的 2×2 橡木与斜枝被识别为一个 exact 树组件，接触的云杉不被吸收；已提交位置
   被替换成另一树种时立即进入 `STALE_TREE`，不因“任意原木”配额继续破坏；
10. 真实 `ActionPack` 在单格宽 L 形走廊经过强制拐角并以同一 request id 到达；
11. exact 橡木配额在整树中途满足后，经 `TaskManager` 的 10 Tick `SAFETY` 暂停/恢复仍保持
   同一 `TreeId`、不增加 `EpisodeMemory` 排除项、完成已提交的 4 根原木，并保持旁边独立树不变。

这 11 个场景已在提交 `9155c9c` 的 CI #106 中随 Fabric 35/35 与 NeoForge 11/11 运行通过。
当前每项使用独立 batch 且保持默认单次尝试；涉及单调放置证据的场景不得直接启用原位 retry，
未来若设置 `attempts`/`requiredSuccesses`，必须同时提供隔离坐标或重建账本/世界的夹具。
第 9 项验证 2×2/斜枝的检测、exact 边界与首次规划，不等同于整棵复杂树已实际采伐完成；
第 11 项直接触发 `TaskManager` 的 `SAFETY` pause owner，并用有界 safety-origin `HoldTask`
模拟接管窗口；它不等同于真实怪物或 `NavSafetyNet` 端到端触发链。不能从上述清单推断这些
更宽行为已通过双加载器验证。

### 非目标

- 不在 P0 重写全部 A*；
- 不迁移所有 Task；
- 不把新增日志当作行为已经变聪明的证明；
- 不使用真实付费 LLM 作为基础回归依赖；
- 不以 NeoForge 普通专服能启动代替 NeoForge GameTest。

### 验收

- 同一失败场景在 Fabric 和 NeoForge 上能稳定复现或稳定通过，不出现加载器专属语义分叉；
- 测试失败报告能指出导航阶段、目标、路径状态和最后进展，而非只给出 `timeout`；
- 动态墙、固定错误站位和振荡场景在修复前能够红灯、修复后能够绿灯；
- `common:test`、Fabric GameTest、NeoForge GameTest server、双端生产构建均有独立结果；
- 两个生产 JAR 均不包含测试入口、测试命令或测试资源；
- 只有干净提交上的完整矩阵结果可以成为 P0 验证证据。

### 依赖

- 无前置阶段；这是后续所有阶段的证据基础。

## 7. P1：交互姿态、路径进度与整树采伐竖切

**状态：实现完成 / 提交 `9155c9c` 的 CI #106 通过；发布 baseline 未 pin**

### 范围

- 修正路径执行器把“处在下一节点旁边”误判为已经到达节点的问题；
- 以目标距离/剩余路径成本下降判断有效进展，识别原地横移、绕圈和 A/B 往返；
- 引入 `InteractionPosePlanner`，枚举合法工作位并比较当前站位、路径成本、视线、风险和连续作业收益；
- 普通交互导航默认非破坏性，不能为了接近树或容器静默挖墙；
- 为原木采集增加“一棵树一次提交”的 `TreeFellingSession` 或等价执行器；
- 有界识别树根、相连原木、自然树证据和剩余原木；
- 在第一次破坏前证明完整组件的每根原木至少有一个干燥、可视且可达的工作姿态；无法完成的
  高树整棵拒绝，不能先砍树根再留下浮空树干；
- 预检工作位与路径不得以本次组件的原木或叶簇作为脚下支撑；每个提交位置锁定原始
  `BlockState`，不同树种替换、旋转或其他世界变更都不能被宽泛原木标签吞掉；
- 技能级脚位约束和永久排除集必须同时进入初始 A*、路线复验、string-pull 与动态重规划；
  不能在规划时拒绝树冠支撑，却在执行器重规划时重新走回同一危险支撑；
- 通过双加载器共同启用的 `BlockItem` 入口记录玩家与 Bot 放置过的原木位置，按维度原子持久化；
  有放置证据的组件一律不能进入自动整树采伐；P1 证据单调只增不减，破坏已记录原木不会自动
  删除坐标。旧坐标造成的保守误拒绝优于因区块/账本非事务保存而丢失来源后误砍玩家结构；
- 损坏、不兼容、非普通文件或缺失的放置账本都会锁定自动整树采伐；普通放置与后续成功写盘
  不能用不完整数据“自愈”。每次可信启动先创建 durable active-session marker；`STOPPING`
  只能预刷账本，只有加载器的终止 `STOPPED` 事件中再次 flush 成功后才能删除标记；崩溃、持续
  写盘失败或未抵达该事件时，下次启动不能从旧合法快照误恢复。`STOPPED` 不被当作区块保存成功

  互相覆盖退避后每 5 Tick 重建会话；

## 8. P2：正式导航契约与多终点路径

**状态：实现完成 / 代码提交 `2450e13` 的 CI #129 通过；发布 baseline 未 pin**

### 范围

- `NavGoal` 以 sealed variants 定义 `Exact`、`Near`、`Interaction`、`FollowRing`、`Flee`、
  `Composite`；构造器约束逃生目标、策略和有界搜索范围，动态目标不进入全局不可达记忆；
- `NavigationRequest`、`NavigationHandle` 与 `NavigationResult` 形成 request-scoped 生命周期，
  明确 `PLANNING`、`FOLLOWING`、`ARRIVED`、`BLOCKED`、`STALE_WORLD`、`PREEMPTED`、
  `CANCELLED`、`FAILED`，并保留 resolved goal、成本、失败分类、世界版本、request id、
  搜索指标和证据作用域，任务不再通过“是否空闲”猜终止结果；
- `MultiGoalAStarPathfinder` 在一个 heading-aware、cost-sensitive frontier 中直接比较多个合法终点，
  保留方向/转弯成本、可采纳启发式和已关闭状态重开，避免逐候选重复同步 A*；
- `NavigationSearchBudget` 对每 Bot、每 Tick 和全服务器使用确定性、可观测预算；搜索耗尽返回
  inconclusive `SEARCH_BUDGET`，不会伪装成真实不可达，也不会写入错误排除记忆；
- 缓存键纳入世界/维度、策略、目标指纹、约束、权限和 world revision；服务端成功
  `Level#setBlock` 后使 A*、多目标和站立性缓存失效，旧路线不能跨世界修改继续命中；
- `ProvenRoute` 保存一次有界 A* 证明的完整路径和后缀；长路线只执行安全中继前缀，后续 segment
  复用同一证明后缀、request id、heading 和累计成本，不为每段新开 frontier；起点合法性、
  goal fingerprint 或 world revision 改变时放弃后缀并重新规划；
- `PathExecutor` 的 string-pull 进度提交对 WALK/DIAGONAL 保持有界容差，对 JUMP/DROP 要求精确，
  DIG/PILLAR 不预提交；动态封路、交互目标失效和危险变化触发保留原约束的有界局部重规划；
- `FollowRing` 按稳定 cadence 跟随移动实体，在刷新窗口间继续执行仍安全的旧路线；动态刷新不耗尽
  失败重规划额度。`Flee` 只在显式有界 escape policy 下运行；
- 旧 `ActionPack` 导航入口通过事务适配器映射到同一 handle/result，不改变 legacy snapped
  interaction endpoint 的语义，也不允许通用走路隐式升级为破坏世界。

P2 当前新增 **14 个** loader-neutral 共享 GameTest：目标 variants、单 frontier 全局最低成本、
交互姿态一次搜索、策略门禁、搜索上限分类、world revision 缓存失效、handle 生命周期、
`ProvenRoute` 中继、动态封路替代终点、过期交互目标、移动 `FollowRing`、有界 `Flee`、同 handle
长路线分段和 legacy `ActionPack` 适配。Fabric 与 NeoForge 使用同一测试体、batch 和阈值；当前

### 非目标

- 不承诺跨维度传送门规划；

### 验收

- 模糊或复合命令才请求 AI，并将返回结果送入权限、schema、事实与可执行性验证门；
- 将大量低层工具收敛为少量高层能力，控制流回到服务器程序；
- AI 可提出 Mission DAG 候选、建筑风格和恢复假设，但不能直接实例化底层动作；
- 对话与行动使用独立 epoch/lease，过期回复不能修改新 Mission；
- API Key 缺失、请求失败或超时不会中止已经验证的 Mission；
- 对模型输入输出记录 token、延迟、provider、模型、decision epoch 和验证结果，且不记录明文密钥。

### 非目标

- 不追求通过增加工具数量表现“更智能”；
- 不让模型原样重试失败工具直到成功；
- 不使用超长提示词模拟任务状态机；
- 不允许模型读取超出当前权限与感知边界的世界状态；
- 不把真实付费模型测试放入每次 PR 的强制门禁。

### 验收

- 明确移动、跟随、采集、暂停、继续和取消命令零 AI 调用；
- 模糊复合命令通常一次模型调用生成合法候选，失败时返回可解释歧义而非乱执行；
- 非法坐标、越权能力、过期目标和未确认建筑提案被验证门拒绝；
- API 中断后任务继续，恢复连接不会重复提交旧动作；
- AI 不能伪造完成、库存或路径结果；
- 中英文 paraphrase 意图回归达到项目发布门槛并记录模型版本。

### 依赖

- P3–P7 已提供稳定、少量且可验证的高层能力契约。

## 15. P9：多 Bot 协作与性能预算

**状态：开放目标 / 未完成**

### 范围

- 持久工单、角色能力、区域租约、资源预留和容器写租约；
- 将团队目标拆为有依赖的子工单，避免重复砍树、重复备料和争抢容器；
- 单 Bot 失败只重排受影响工单，不重启整个团队 Mission；
- 路径搜索使用世界快照或分时预算，主线程只应用已验证结果；
- `TpsGuard` 管理全局扫描、路径、感知、建筑和 AI 回调预算；
- 将原木来源证据改为有界追加日志/批量压缩或等价的 crash-safe 方案，避免木结构施工时每个 dirty
  Tick 在主线程重写并 fsync 全量 JSON；性能优化不能削弱损坏/崩溃后的 fail-closed 语义；只有
  获得可证明的世界代际/区域所有权事务后，压缩器才能回收陈旧坐标；
- 为单 Bot 与多 Bot 建立公平调度，防止一个大型任务饿死其他 Bot；
- 性能报告绑定参考硬件、JVM 参数、视距、种子、Bot 数和任务组合。

### 非目标

- 不做跨服务器分布式 Bot 集群；
- 不在单 Bot 契约不稳定时提前增加团队复杂度；
- 不允许多个 Bot 同时拥有同一区域或同一物资的排他租约；
- 不用降低正确性验证来换取表面 TPS。

### 验收

- 4 个 Bot 同时执行采集/生产/建筑混合任务 30 分钟，在约定参考环境 TPS `>= 19`；
- 相同树、箱子、熔炉和建筑工作区不会被重复认领；
- Bot 失联、死亡或任务取消后租约和预留能在有界时间释放；
- 全局路径/扫描预算超限时任务降频而非服务器长卡；
- 团队完成状态由所有子工单后置条件共同决定；
- 单 Bot 与多 Bot 结果在 Fabric/NeoForge 服务端语义一致。

### 依赖

- P2 导航预算；
- P3 工单可组合契约；
- P4 世界事实；
- P5–P8 稳定的业务与 AI 高层接口。

## 18. 阶段状态变更规则

阶段状态只能按以下顺序变化：

```text
开放目标
  → 实现中
  → 待单元/编译验证
  → 待双加载器运行验证
  → 提交绑定 CI 通过 / 待显式 pin
  → VERIFIED 发布基线
```

若出现以下任一情况，应退回前一状态：

- 只在一个加载器通过；
- 测试运行于脏工作树但报告未绑定差异；
- 测试使用了与发布不同的 profile、权限或版本；
- 行为只能通过 operator 特权完成，却被当作 strict 结果；
- 场景没有权威后置条件，只依据聊天或任务自报成功；
- 新提交改变相关代码但旧 evidence 未重新运行；
- 依赖许可证、迁移或安全审查尚未完成。
